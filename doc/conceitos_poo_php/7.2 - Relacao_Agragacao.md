## Definição formal (engenharia de software)

Agregação é uma relação todo–parte fraca, onde:

- Existe relação “tem um”

- As partes podem existir independentemente

- O todo não controla o ciclo de vida das partes

- As partes podem ser compartilhadas

- A destruição do todo não destrói as partes

Formalmente:

- Existe agregação quando A “tem” B,
- mas B não depende existencialmente de A.

## Posição acadêmica importante (ponto crítico)

Em engenharia moderna, especialmente:

- Clean Architecture

- DDD

- SOLID

👉 Agregação NÃO é implementada como algo distinto da associação no código.

Ela é uma decisão semântica de modelagem, não um mecanismo técnico.

📌 UML diferencia:

- Associação

- Agregação (losango vazio)

- Composição (losango cheio)

📌 Código NÃO diferencia associação de agregação

## Comparação científica direta: 

| Critério            | Associação | Agregação        | Composição       |
| ------------------- | ---------- | ---------------- | ---------------- |
| Tipo                | Genérica   | Todo–parte fraca | Todo–parte forte |
| Dependência de vida | ❌          | ❌                | ✅                |
| Criação interna     | ❌          | ❌                | ✅                |
| Compartilhável      | ✅          | ✅                | ❌                |
| “É parte de”        | ❌          | ✅                | ✅                |
| Controle do todo    | ❌          | Parcial          | Total            |

## Agregação vs Associação (diferença real)

Essa é a pergunta mais importante:

- Se agregação e associação usam o mesmo código, por que agregação existe?

Resposta:

- 👉 Para expressar significado de domínio, não técnica.

## Exemplo conceitual clássico

Universidade e Professor

- Universidade tem professores

- Professor não pertence à universidade

- Professor pode trocar de universidade

- Professor existe sem universidade

👉 Agregação

## Exemplo de Agregação em PHP:

```php
class Professor
{
    public function __construct(
        private string $nome
    ) {}
}

class Universidade
{
    /** @var Professor[] */
    private array $professores;

    public function __construct(array $professores)
    {
        $this->professores = $professores;
    }
}
```

Análise científica

- Universidade não cria Professor

- Recebe instâncias prontas

- Professores podem ser reutilizados

- Não há controle de vida

👉 Agregação legítima

## Agregação NÃO É “array de objetos”

Isso é um erro conceitual comum.

```php
class Pedido {
    private array $itens;
}
```

Isso não define agregação.

A decisão está no significado:

- Quem criou?

- Quem destrói?

- Pode existir fora?

## Agregação no mundo real (casos legítimos)

Exemplos corretos:

- Time → Jogadores

- Empresa → Funcionários

- Curso → Alunos

- Biblioteca → Livros

- Pedido → Produto (produto ≠ item)

📌 Observe:

- Produto existe fora do pedido

- Pedido apenas referencia

## Comparação: Agregação vs Composição (caso sutil)

Pedido + Produto (agregação)

```php
class Pedido {
    private array $produtos;
}
```

Pedido + ItemPedido (composição)

```php
class Pedido {
    private array $itens;
}
```

- Produto é agregado

- ItemPedido é composto

Esse é um padrão clássico de sistemas comerciais

## Regra de ouro definitiva

Faça estas perguntas:

- Isso é parte estrutural?

- Pode existir sozinho?

- Pode ser compartilhado?

- Quem controla o ciclo de vida?

Resultado:

- Sim / Sim / Sim / Externo → Agregação

- Não / Sim / Sim / Externo → Associação

- Não / Não / Não / Interno → Composição
